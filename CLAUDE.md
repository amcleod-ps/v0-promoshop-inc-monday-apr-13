# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

PromoShop Inc.'s marketing site: Next.js 16 (App Router) + React 19 on Vercel,
backed by Supabase Postgres + Storage. Originally generated by v0.app. Content
(images, copy, brands, products, team, theme colours) is editable at runtime
without code changes or redeploys.

## Commands

```bash
pnpm install
pnpm dev      # dev server on :3000
pnpm build    # production build — also the reliable type-check (tsc noEmit via Next)
pnpm start    # serve the production build
pnpm lint     # eslint .

# Regenerate supabase/migrations/0003_seed_data.sql from in-repo seed files.
# Run after editing lib/seed-data/* or lib/cms/team.ts.
pnpm tsx scripts/generate-seed-sql.ts
```

There is no test framework configured. `pnpm build` is the de-facto correctness
gate (it type-checks). The package is named `my-project` in package.json; the
real name is PromoShop Inc.

## The one architectural idea to understand first

Every page renders with `export const dynamic = 'force-dynamic'` (set in
`app/layout.tsx`), so **the site fetches fresh data on every request**. On top
of that, image/file URLs are cache-busted with `?v=<row.updated_at>`. Together
these mean: edit a row in Supabase (via the Table Editor *or* `/admin-dashboard`)
and the change is live on the next request — through browser, CDN, and
`next/image` caches — with no rebuild. Don't add static generation, `revalidate`
timers, or `next/image` optimization to content paths; it breaks this contract.

## Two-layer data model: static seed = fallback, Supabase = live override

Content has two sources that must stay in sync:

- **`lib/seed-data/*.ts`** (`brands.seed.ts`, `products.seed.ts`) and
  **`lib/cms/team.ts`** are the compiled-in source of truth. `lib/brands.ts`
  and `lib/products.ts` just re-export them (plus helpers). Client components
  import these for **types** and as a **static fallback** when Supabase is
  unreachable. `/my-quote` reads the static `PRODUCTS` directly.
- **`lib/supabase/*.ts`** fetch the **live** data. Server Components use these:
  homepage (`getHeroSlides`, `getSupabaseBrands`), `/studio` (`getAllProducts`),
  `/brands/[slug]` (`getSupabaseBrandBySlug` + `getAllProducts`), etc.
- **`scripts/generate-seed-sql.ts`** compiles the seed files + team roster into
  `0003_seed_data.sql`. **The seed files are the only hand-edited input to that
  migration — never edit `0003_seed_data.sql` by hand.** After changing seeds,
  rerun the generator so a fresh DB seeds with the same defaults the static
  fallback shows.

## The "registry → provider → resolver with fallback" CMS pattern

Four parallel registries follow the *same* shape. Learn one and you know all
four. The root layout `Promise.all`-fetches each map, wraps children in a
context provider, and components read through a resolver that falls back to a
hard-coded default and applies `?v=<updated_at>` cache-busting:

| Concern | Supabase table | Fetch (server) | Provider | Read API |
| --- | --- | --- | --- | --- |
| Images | `site_images` | `getSiteImagesMap` | `SiteImagesProvider` | `<SiteImage imageId defaultSrc>` / `useSiteImageUrl` (client); `resolveSiteImageUrl` (server) |
| Text | `site_content` | `getSiteContentMap` | `SiteContentProvider` | `useSiteText(key, fallback)` (client); `resolveSiteText(map, …)` (server/loops) |
| Team | `team_members` | `getTeamMembers` | `TeamMembersProvider` | `useTeamMembers()` (see below) |
| Theme | `site_theme` | `getSiteThemeMap` | — (CSS in `<head>`, see below) | CSS class overrides + `var(--brand-*)` |

Conventions when extending this:
- **Keys are stable dotted strings**: `site.logo`, `brand.<slug>.logo`,
  `brand.<slug>.lifestyle`, `team.<slug>`, `home.slideshow.<n>`. Helpers live in
  `lib/image-registry.ts` (`slugify`, `teamImageId`, `brandLogoId`,
  `brandLifestyleId`).
- **Every `lib/supabase/*` getter swallows errors / missing env vars** and
  returns `{}`/`[]`/defaults. This is deliberate: a Supabase outage or unset
  env var must never 500 the layout (which would take down every route,
  including `/admin-dashboard`). Preserve this defensive shape in new getters.
- A new editable image: insert a `site_images` row (`key`, `label`, `url`,
  `alt_text`), then render `<SiteImage imageId="your.key" defaultSrc="…">`.

### Team roster has two code paths

`useTeamMembers()` (`hooks/use-team-members.ts`) prefers the `team_members`
table when it has rows (migration 0005+), otherwise falls back to static
`TEAM_MEMBERS` with `site_content` text overrides. DB members set
`directImageUrl`, which `TeamMemberAvatar` prefers over the
`site_images['team.<slug>']` registry — otherwise the seeded default photo
shadows a freshly uploaded one.

### Theme override (the trickiest part)

The site styles with Tailwind v4 arbitrary-value classes like
`text-[#ef473f]`, `bg-[#ef473f]/20`, `hover:border-[#ef473f]`.
`themeOverrideCss()` in `lib/supabase/theme.ts` generates `!important` CSS that
targets the **exact escaped class selectors** Tailwind emits — across state
variants (hover/focus/active/group-*) and opacity modifiers — for each brand
colour the admin changed. The layout injects this in `<head>` so it paints
before utilities, avoiding a flash of the original palette. Do **not** simplify
to attribute-contains selectors (`[class*="bg-[#ef473f]"]`): they'd wrongly
match `hover:`/opacity variants and force elements into hover state at load.

## Supabase clients — pick the right one

- `lib/supabase/server.ts` — Server Components, anon key, cookie-aware (`@supabase/ssr`). Default for reads.
- `lib/supabase/client.ts` — browser, anon key.
- `lib/supabase/admin.ts` — **service-role key, bypasses RLS, `server-only`**.
  Use *only* in `/admin-dashboard` server actions. Never import it from a
  `"use client"` file or anything reaching the browser bundle.

## `/admin-dashboard` — the live CMS

Despite README phrasing about "no admin UI", `app/admin-dashboard/` **is** a
working CMS (image replace/remove, text editing, brand/hero/product/team create,
theme editor). It is **intentionally unauthenticated** — the URL is the secret
(robots-disallowed, unlinked). Its server actions live in `actions.ts` and
`create-actions.ts`, use the admin (service-role) client, validate a 10 MB image
cap, and call `revalidatePath("/", "layout")` to push changes live.

**Gotcha — brand logos are written to two places.** A brand logo is read from
both `brands.logo_url` (canonical) and `site_images['brand.<slug>.logo']`
(legacy override slot, checked first by `useImageSrc`). `replaceImage` /
`removeImage` update *both*; if you touch one, update the other or the override
silently shadows the canonical column.

Server Actions accept up to 10 MB bodies (`serverActions.bodySizeLimit` in
`next.config.mjs`) to match the uploader; the default 1 MB would reject larger
images with an opaque error.

## Client-only state (not real security)

Customer "auth" (`lib/auth/AuthProvider.tsx`, `/sign-in`, `/sign-up`) and the
quote cart (`lib/quote-context.tsx`) are **localStorage-only** — UI
personalization, not access control. Locale (`lib/locale-context.tsx`, CAN/USA
spelling via `t(key)`) is likewise localStorage-backed and hydrates after first
render to keep SSR markup stable. The only data that actually persists server-
side is quote-request submissions (`app/actions/quotes.ts` → `quote_requests`,
Zod-validated). When the Resend env vars are set, each successful submission
also fires a best-effort staff notification email
(`lib/email/quote-notification.ts`, called via `after()` so it never blocks or
fails the submission; with the vars unset it silently no-ops).

## Conventions

- **Tailwind v4, CSS-first**: config lives in `app/globals.css` (`@theme`), not
  a `tailwind.config.js`. `components.json` (shadcn/ui "new-york", RSC) drives
  generation; UI primitives are in `components/ui/`.
- Path alias `@/*` → repo root.
- `next/image` runs with `unoptimized: true` globally; new external image hosts
  must be added to `images.remotePatterns` in `next.config.mjs`.
- Migrations in `supabase/migrations/` are applied **in order, by hand** via the
  Supabase SQL Editor (0001 → 0005). There are five (README mentions three);
  0004 adds `site_content`, 0005 adds `team_members` + `site_theme`.

## Environment

Copy `.env.example` → `.env.local`. `NEXT_PUBLIC_SUPABASE_URL` and
`NEXT_PUBLIC_SUPABASE_ANON_KEY` are required for the public site (the
`NEXT_PUBLIC_` prefix is mandatory). `SUPABASE_SERVICE_ROLE_KEY` is server-only
and required only for `/admin-dashboard` writes — never give it the
`NEXT_PUBLIC_` prefix. Optional, server-only: `RESEND_API_KEY`,
`QUOTE_NOTIFICATION_EMAIL`, `QUOTE_NOTIFICATION_FROM` enable quote-request
email notifications (`docs/RESEND-EMAIL-SETUP.md` is the client-facing setup
guide; `docs/ADMIN-LOGIN-SETUP.md` covers admin-dashboard access).
