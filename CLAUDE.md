# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

PromoShop Inc.'s marketing site: Next.js 16 (App Router) + React 19 on Vercel,
backed by Supabase Postgres + Storage. Originally generated by v0.app. Content
(images, copy, brands, products, team, theme colours) is editable at runtime
without code changes or redeploys.

## Commands

```bash
pnpm install
pnpm dev      # dev server on :3000
pnpm build    # production build — also the reliable type-check (tsc noEmit via Next)
pnpm start    # serve the production build
pnpm lint     # eslint . (flat config in eslint.config.mjs; ignores components/ui)

# Regenerate supabase/migrations/0003_seed_data.sql from in-repo seed files.
# Run after editing lib/seed-data/* or lib/cms/team.ts.
pnpm tsx scripts/generate-seed-sql.ts
```

There is no test framework configured. `pnpm build` is the de-facto correctness
gate (it type-checks); `pnpm lint` must also stay green — run both before
pushing. `react-hooks/set-state-in-effect` is deliberately off in
eslint.config.mjs: the localStorage hydration pattern below and the
dashboard's preview/re-sync effects are intentional setState-in-effect.

## The one architectural idea to understand first

Every page renders with `export const dynamic = 'force-dynamic'` (set in
`app/layout.tsx`), so **the site fetches fresh data on every request**. On top
of that, image/file URLs are cache-busted with `?v=<row.updated_at>` (always
via `withCacheBust` in `lib/cache-bust.ts`, which appends with `&` when the
URL already has a query string — never hand-roll the join). Together
these mean: edit a row in Supabase (via the Table Editor *or* `/admin-dashboard`)
and the change is live on the next request — through browser, CDN, and
`next/image` caches — with no rebuild. Don't add static generation, `revalidate`
timers, or `next/image` optimization to content paths; it breaks this contract.

## Two-layer data model: static seed = fallback, Supabase = live override

Content has two sources that must stay in sync:

- **`lib/seed-data/*.ts`** (`brands.seed.ts`, `products.seed.ts`) and
  **`lib/cms/team.ts`** are the compiled-in source of truth. `lib/brands.ts`
  and `lib/products.ts` just re-export them (plus helpers). Client components
  import these for **types** and as a **static fallback** when Supabase is
  unreachable. `/my-quote` is a server page passing `getAllProducts()` (with
  the static fallback) into its client component.
- **`lib/supabase/*.ts`** fetch the **live** data. Server Components use these:
  homepage (`getHeroSlides`, `getSupabaseBrands`), `/studio` (`getAllProducts`),
  `/brands/[slug]` (`getSupabaseBrandBySlug` + `getAllProducts`), `/my-quote`
  (`getAllProducts` for the manual add-product picker), etc.
- **`scripts/generate-seed-sql.ts`** compiles the seed files + team roster into
  `0003_seed_data.sql`. **The seed files are the only hand-edited input to that
  migration — never edit `0003_seed_data.sql` by hand.** After changing seeds,
  rerun the generator so a fresh DB seeds with the same defaults the static
  fallback shows.

## The "registry → provider → resolver with fallback" CMS pattern

Four parallel registries follow the *same* shape. Learn one and you know all
four. The root layout `Promise.all`-fetches each map, wraps children in a
context provider, and components read through a resolver that falls back to a
hard-coded default and applies `?v=<updated_at>` cache-busting:

| Concern | Supabase table | Fetch (server) | Provider | Read API |
| --- | --- | --- | --- | --- |
| Images | `site_images` | `getSiteImagesMap` | `SiteImagesProvider` | `<SiteImage imageId defaultSrc>` / `useSiteImageUrl` (client); `resolveSiteImageUrl` (server) |
| Text | `site_content` | `getSiteContentMap` | `SiteContentProvider` | `useSiteText(key, fallback)` (client); `resolveSiteText(map, …)` (server/loops) |
| Team | `team_members` | `getTeamMembers` | `TeamMembersProvider` | `useTeamMembers()` (see below) |
| Theme | `site_theme` | `getSiteThemeMap` | — (CSS in `<head>`, see below) | CSS class overrides + `var(--brand-*)` |

Conventions when extending this:
- **Keys are stable dotted strings**: `site.logo`, `brand.<slug>.logo`,
  `brand.<slug>.lifestyle`, `team.<slug>`, `home.slideshow.<n>`. Helpers live in
  `lib/image-registry.ts` (`slugify`, `teamImageId`, `brandLogoId`,
  `brandLifestyleId`).
- **Every `lib/supabase/*` getter swallows errors / missing env vars** and
  returns `{}`/`[]`/defaults. This is deliberate: a Supabase outage or unset
  env var must never 500 the layout (which would take down every route,
  including `/admin-dashboard`). Preserve this defensive shape in new getters.
- A new editable image: insert a `site_images` row (`key`, `label`, `url`,
  `alt_text`), then render `<SiteImage imageId="your.key" defaultSrc="…">`.
- **Editable text without a seeded row**: `lib/cms/text-slots.ts` registers
  copy that previously shipped hard-coded (studio/quote headings, header CTA,
  footer column headings…). The dashboard's Text tab merges these slots in as
  empty editors; saving upserts a real `site_content` row, and blank falls
  back to the compiled-in copy. Components read the same key with
  `textFallback(key)` as the fallback so registry and renderer can't drift —
  add new always-editable strings there, not as bare literals.
- **Inline copy formatting**: `lib/rich-text.tsx` renders a SMALL, XSS-safe
  subset of Markdown — `**bold**`, `*italic*`/`_italic_`, `[text](url)` — to
  React nodes (never `dangerouslySetInnerHTML`; link targets gated through
  `isSafeLinkTarget`, so a Table-Editor value can't inject markup). It is
  *opt-in per slot*: a renderer calls `renderInlineMarkdown(value)` (inline,
  caller owns the `<p>`) or `<RichText value>` (multi-line). Adopted so far on
  the About body paragraphs (`app/about/page.tsx`) and the footer tagline + ADA
  notice; other slots stay literal until their renderer opts in (so `**` in a
  CTA label shows as asterisks). Block controls (headings/lists) are
  deliberately withheld — admin headings would break the a11y heading order.
  This default control set is pending Victor's confirmation.
- **Image display mode (cover/contain)**: `lib/image-fit.ts`. Placements with
  a fixed-aspect frame (hero slides, `about.hero`, brand lifestyle backdrops)
  read a `site_content` row keyed `image-fit.<slot>` (slot = `hero_slide.<id>`
  or the `site_images` key) and map `contain` → `object-contain`, anything
  else → `object-cover`. The dashboard's Images tab shows the selector only
  for slots whose renderer consults it (`updateImageFit` server action);
  values are sanitized on read because Table-Editor edits bypass validation.
- **Image display size (sm/md/lg)**: `lib/image-size.ts`, the companion to
  image-fit for *free-standing* images (where fit controls cropping, size
  scales the whole image). Reads a `site_content` row keyed `image-size.<slot>`
  and maps `sm`/`lg` to a placement-specific height class, anything else → the
  default `md`. Today it drives `site.logo` (the header reads it in
  `components/header.tsx`, the footer in `components/footer.tsx`, each scaling
  from its own base via `pickBySize`); the dashboard Images tab shows the
  selector only for `site.logo` (`updateImageSize` server action,
  `sizeSlotForSiteImage`). Same sanitize-on-read contract as image-fit; add a
  slot by giving its renderer a `pickBySize` height map and extending
  `sizeSlotForSiteImage`.
- **Squarespace resolution hints**: seeded product images carry
  `?format=NNNNw` (the CDN's served width). `withMinImageWidth` in
  `lib/image-resolution.ts` raises the hint per render context (cards 750w,
  modal 1500w, lightbox 2500w) so low seeded hints don't upscale soft;
  non-Squarespace URLs pass through untouched. It never lowers an explicit
  higher hint and preserves the `?v=` cache-bust param.

### Team roster has two code paths

`useTeamMembers()` (`hooks/use-team-members.ts`) prefers the `team_members`
table when it has rows (migration 0005+), otherwise falls back to static
`TEAM_MEMBERS` with `site_content` text overrides. DB members set
`directImageUrl`, which `TeamMemberAvatar` prefers over the
`site_images['team.<slug>']` registry — otherwise the seeded default photo
shadows a freshly uploaded one.

### Theme override (the trickiest part)

The site styles with Tailwind v4 arbitrary-value classes like
`text-[#ef473f]`, `bg-[#ef473f]/20`, `hover:border-[#ef473f]`.
`themeOverrideCss()` in `lib/supabase/theme.ts` generates `!important` CSS that
targets the **exact escaped class selectors** Tailwind emits — across state
variants (hover/focus/active/group-*, with hover rules gated behind
`@media (hover: hover)` like Tailwind v4) and opacity modifiers — for each
brand colour the admin changed. Values that aren't plain hex colours are
discarded before injection (Table-Editor edits bypass the dashboard's
validation, and the CSS lands in a `<style dangerouslySetInnerHTML>`). The layout injects this in `<head>` so it paints
before utilities, avoiding a flash of the original palette. Do **not** simplify
to attribute-contains selectors (`[class*="bg-[#ef473f]"]`): they'd wrongly
match `hover:`/opacity variants and force elements into hover state at load.

## Supabase clients — pick the right one

- `lib/supabase/server.ts` — Server Components, anon key, cookie-aware (`@supabase/ssr`). Default for reads.
- `lib/supabase/client.ts` — browser, anon key.
- `lib/supabase/admin.ts` — **service-role key, bypasses RLS, `server-only`**.
  Use *only* in `/admin-dashboard` server actions. Never import it from a
  `"use client"` file or anything reaching the browser bundle.

## `/admin-dashboard` — the live CMS

`app/admin-dashboard/` **is** a working CMS (image replace/remove + size/fit,
text editing, brand/hero/product/team/collection create, product tags, theme
editor). By default it is **open to anyone who has the URL** — the URL is the secret
(noindex-metadata'd, unlinked; deliberately NOT named in robots.txt —
generated by `app/robots.ts` — which would advertise the path). An **opt-in HTTP Basic gate** exists: setting
`ADMIN_DASHBOARD_PASSWORD` makes `proxy.ts` challenge the page and
`requireAdminAction()` (`lib/admin-auth-action.ts`) re-verify every server
action (actions are invocable from any route by Next-Action id, so the proxy
matcher alone can't protect them); unset preserves the open mode. Its server
actions live in `actions.ts` and
`create-actions.ts`, use the admin (service-role) client, validate a 10 MB image
cap, and call `revalidatePath("/", "layout")` to push changes live. Return
errors via `adminActionError` (`lib/admin-error.ts`): raw Postgres/Storage
messages are logged server-side and never surfaced to the dashboard (those
actions are reachable by Next-Action id even with the gate off), while
validation/duplicate messages stay specific.

**Gotcha — brand logos are written to two places.** A brand logo is read from
both `brands.logo_url` (canonical) and `site_images['brand.<slug>.logo']`
(legacy override slot, checked first by `useImageSrc`). `replaceImage` /
`removeImage` update *both*; if you touch one, update the other or the override
silently shadows the canonical column.

Server Actions accept up to 10 MB bodies (`serverActions.bodySizeLimit` in
`next.config.mjs`) to match the uploader; the default 1 MB would reject larger
images with an opaque error.

## Client-only state (not real security)

Customer "auth" (`lib/auth/AuthProvider.tsx`, `/sign-in`, `/sign-up`) is a
browser-local quote profile only — no password is collected, no server account
is created, and it must never be described as access control. The quote cart
(`lib/quote-context.tsx`) is also **localStorage-only**. Locale (`lib/locale-context.tsx`, CAN/USA
spelling via `t(key)`) is likewise localStorage-backed and hydrates after first
render to keep SSR markup stable. The only data that actually persists server-
side is quote-request submissions (`app/actions/quotes.ts` → `quote_requests`,
Zod-validated with per-field length caps, a hidden honeypot field both public
forms send, and a per-IP in-memory rate limit — migration 0007's CHECK
constraints and migration 0011's forced ids + email throttle backstop direct
PostgREST inserts). When the Resend env vars are set, each successful submission
also fires a best-effort staff notification email
(`lib/email/quote-notification.ts`, called via `after()` so it never blocks or
fails the submission; with the vars unset it silently no-ops).

## Conventions

- **Tailwind v4, CSS-first**: config lives in `app/globals.css` (`@theme`), not
  a `tailwind.config.js`. `components.json` (shadcn/ui "new-york", RSC) drives
  generation; UI primitives are in `components/ui/`.
- Path alias `@/*` → repo root.
- `next/image` runs with `unoptimized: true` globally; new external image hosts
  must be added to `images.remotePatterns` in `next.config.mjs`.
- Migrations in `supabase/migrations/` are applied **in order, by hand** via the
  Supabase SQL Editor (0001 → 0011). There are eleven; 0004 adds `site_content`,
  0005 adds `team_members` + `site_theme`, 0006 adds the `assign_sort_order`
  insert triggers the dashboard's create actions rely on for race-free
  ordering (they fall back to read-max+1 without it), 0007 adds
  `quote_requests` CHECK length constraints backstopping direct PostgREST
  inserts, 0008 forces server-side timestamps on `quote_requests` inserts
  (plus a guarded seeded-copy fix), 0009 adds the `products.tags` column
  (US/Canada toggle + forgiving filter tags, Priority 3), and 0010 adds
  `collections` + `collection_products` (Priority 4 Collections; reads in
  `lib/supabase/collections.ts` are defensive so the site is unaffected until
  it's applied). 0011 forces quote ids server-side, adds a small DB-side email
  throttle for direct inserts, and fixes NULL-scoped product-image ordering.
  0003 is reseed-safe:
  conflicts are DO NOTHING / no-op, so re-running never overwrites admin edits.
  **Resilience contract:** code that reads a column added by a not-yet-applied
  migration must do so in a SEPARATE query that fails soft (see
  `getAllProducts`' tags read), so a pre-migration DB degrades gracefully
  instead of collapsing the whole page to the static fallback.

## Environment

Copy `.env.example` → `.env.local`. `NEXT_PUBLIC_SUPABASE_URL` and
`NEXT_PUBLIC_SUPABASE_ANON_KEY` are required for the public site (the
`NEXT_PUBLIC_` prefix is mandatory). `SUPABASE_SERVICE_ROLE_KEY` is server-only
and required only for `/admin-dashboard` writes — never give it the
`NEXT_PUBLIC_` prefix. Optional, server-only: `ADMIN_DASHBOARD_PASSWORD`
enables the admin-dashboard Basic-auth gate (unset = open); `RESEND_API_KEY`,
`QUOTE_NOTIFICATION_EMAIL`, `QUOTE_NOTIFICATION_FROM` enable quote-request
email notifications (`docs/RESEND-EMAIL-SETUP.md` is the client-facing setup
guide; `docs/ADMIN-LOGIN-SETUP.md` covers admin-dashboard access).
