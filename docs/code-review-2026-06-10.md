# Pre-launch review — PromoShop Inc.

**How this was produced:** full fresh read of the committed code on this branch by five parallel reviewers (public pages, admin CMS, data layer/migrations, shared components, config/infra), with the top findings re-verified by me directly. `pnpm build` passes clean (type-check included); regenerating `0003_seed_data.sql` produces a byte-identical file (no seed drift). Items needing a client-supplied value are marked **CLIENT**.

## Launch blockers

**1. The homepage hero slideshow displays cropped brand logos as its "slides."** `lib/cms/home.ts:33-38` and the seeded `hero_slides` rows both point at `/images/mainmemory/1.png`–`4.png` — which are *also* the seeded logos for Patagonia, Peter Millar, Helly Hansen, etc. in `lib/seed-data/brands.seed.ts:20,128,146,191`. They render full-bleed with `object-cover` in a 500px panel (`components/home/hero-slideshow.tsx:48`), so the first thing every visitor sees is a zoomed, cropped Patagonia wordmark rotating into other third-party logos. Same conflict on the About page: `11.png` is both the Rhone logo and the About hero (`scripts/generate-seed-sql.ts:120`). In-code comments admit the ordering was "a best guess" pending client confirmation. **Fix:** real lifestyle/product imagery in `HOME_CONTENT.slideshow` and the about hero, regenerate the seed SQL, give the slides meaningful titles (the title becomes alt text). **CLIENT** must supply/confirm the four hero images and the About hero.

**2. The brand fonts load but are never applied — the whole site renders in system fallback fonts.** `app/layout.tsx:21-37` loads Montserrat, Bebas Neue and DM Sans via `next/font` exposed only as CSS variables, but `app/globals.css` (lines 42-45, 127-144) references literal family names (`font-family: 'Montserrat'`, `'Bebas Neue'`, …). `next/font` registers scoped names, so the literals never match and there's no Google Fonts `<link>` fallback — all typography silently degrades and the webfonts are downloaded for nothing. **Fix:** switch globals.css to `var(--font-montserrat)`, `var(--font-bebas)`, `var(--font-dm-sans)`. Related: `.font-visby` ('Visby CF', used across body text) is a TODO at `globals.css:43,142` — ship the licensed font files or standardize on DM Sans (**CLIENT** decision, commercial font); `--font-mono: 'DM Sans'` is also wrong.

**3. `public/robots.txt` publishes the secret URL of the unauthenticated admin CMS.** The whole `/admin-dashboard` security model is "the URL is the secret," yet `robots.txt:2` (`Disallow: /admin-dashboard`) names it in the one file every scanner fetches first — and `docs/hardening-status.md:43` notes the repo is public, so it leaks there too. Anyone who reads it gets full service-role-backed write access to every image, text block, brand, product, and theme on day one. The auth deferral is on record as a client decision (2026-06-01), but this leak is independent and trivially removable. **Fix:** delete the Disallow line (the page already sets `robots: noindex,nofollow` in its own metadata, which suffices); separately re-raise gating the route (Vercel Deployment Protection or ~30 lines of basic-auth middleware) — **CLIENT** decision to un-defer.

**4. The cache-buster corrupts every seeded product image URL.** `busted()` in `lib/supabase/products.ts:52-55` blindly appends `?v=<updated_at>`, but 43 seeded `product_images` URLs already carry `?format=1500w` (e.g. `products.seed.ts:30`), producing `…png?format=1500w?v=…` — a malformed double query string on the live `/studio` and `/brands/[slug]` catalog the moment migrations are applied. Squarespace's CDN at best ignores the broken param (serving oversized originals), at worst errors. The same naive join exists in `lib/supabase/data.ts:32`, `team.ts:14`, `images.ts:61`, and `components/site-images-provider.tsx:30` — any URL with an existing query string breaks identically. **Fix:** join with `url.includes("?") ? "&" : "?"` in all five helpers.

**5. Privacy Policy and Terms of Service don't exist, yet sign-up requires consenting to them.** Footer links Privacy / Terms / Shipping Policy all go to `"#"` (`components/footer.tsx:183-200`), and `app/sign-up/page.tsx:148` makes users tick "I agree to the Terms of Service and Privacy Policy" where both links are `"#"`. The site collects names, emails, phone numbers (quote requests persist server-side), so this is a legal/compliance gap, not just polish. **Fix:** stub `/privacy`, `/terms`, `/shipping` routes and link them. **CLIENT** must supply the actual policy copy.

## High — broken core flows and admin/data integrity

**6. The sign-up → quote autofill funnel is broken and destroys the user's data.** `app/sign-up/page.tsx:51` writes `promoshop_quote_contact` straight to localStorage, but `QuoteProvider` (root layout) read that key once on mount and survives the client-side `router.push` (`lib/quote-context.tsx:111-131`) — so the contact fields render empty on `/my-quote`, and the first keystroke in any field persists the stale in-memory state, silently overwriting what sign-up saved. **Fix:** call `useQuote().setContactInfo(...)` instead of raw localStorage.

**7. The add-to-quote sign-up gate throws away the user's selections.** In `components/studio/product-detail-modal.tsx:122-136`, an unauthenticated user who picks colours/sizes and clicks "Add N items to Quote" is bounced to sign-up *without* the items being added; after the 7-field form they land on an empty quote. Line 129 also writes `sessionStorage["promoshop_post_auth_redirect"]`, which nothing ever reads — dead code whose comment describes behavior that doesn't exist. **Fix:** add the items to the (non-auth-gated localStorage) cart before redirecting; delete the dead write.

**8. Hero-slide subtitle/CTA fields are editable in the admin but never rendered.** `components/home/hero-slideshow.tsx` renders only image + `bg_color`; `title`/`subtitle`/`cta_text`/`cta_url` mapped in at `app/page.tsx:30-38` are dropped. The dashboard explicitly promises the opposite ("Leave subtitle, CTA label, and CTA URL blank to hide the button…", `dashboard-list.tsx:321`) and the Add-slide form collects CTA fields. An admin will type CTA copy, save successfully, and see nothing. Also latent: `updateHeroSlideText` accepts any string for `cta_url` (no scheme check) — a stored-XSS foothold the day the overlay is built. **Fix:** render the title/subtitle/CTA overlay (or strip the fields from the admin), and validate `cta_url` against `/^(\/|https?:\/\/)/` either way.

**9. The admin "Site images" section bypasses the brand-logo dual-write invariant.** The dashboard lists every `site_images` row — including the 20+ `brand.<slug>.logo` rows — with `target="site_images"` (`app/admin-dashboard/page.tsx:154-157`, `dashboard-list.tsx:214-219`), which updates only the legacy override, not `brands.logo_url`. The two stores desync (public site shows the new logo, the Brand-logos section shows the old one), and clicking "Delete this image slot" on such a row reverts the public site to the stale logo with a misleading confirm message. The proper `target="brand"` path does the dual-write correctly — the dashboard just offers a second path that doesn't. **Fix:** filter `brand.*.logo` (and `team.*`) keys out of the generic section, or route them to `target="brand"`.

**10. Admin team photo/text editors silently do nothing.** Once migration 0005 is applied, `useTeamMembers()` takes the DB branch and ignores `site_content` `team.<slug>.*` overrides (`hooks/use-team-members.ts:25-39`), and `TeamMemberAvatar` prefers `team_members.image_url` (seeded to `/placeholder-user.jpg`) over the `site_images['team.<slug>']` registry. Yet the dashboard still lists 4 team image slots and 12 team text rows; editing any shows "Saved. Live on the site." with zero public effect. **Fix:** hide `team.*` keys from the Images/Text tabs when `team_members` is populated, or route those editors to the `team_members` table. Note the seeded photos are placeholders — **CLIENT** team photos must be uploaded via the Team tab (the path that works).

**11. Creating a hero slide instantly publishes a blank frame to the homepage.** `create-actions.ts:365-377` inserts with `image_url: null, is_active: true`; the slideshow then rotates through an empty frame every 5 seconds until the admin uploads an image. **Fix:** insert inactive and flip on first upload, or skip slides with no image and no `bg_color` at render.

**12. Re-applying the seed migration corrupts live content — the documented maintenance workflow is unsafe.** Three separate hazards: (a) `hero_slides` inserts use `gen_random_uuid()` + `on conflict do nothing` with no natural unique key (`scripts/generate-seed-sql.ts:84-93`), so every re-run adds 4 duplicate active slides — despite the file's header claiming idempotency; (b) the generated 0003 does `delete from product_colours where product_sku=…` before re-inserting, which cascades to `product_images` and wipes every admin-added colour and uploaded gallery image on seeded SKUs; (c) `0005_team_and_theme.sql:75-80` upserts `role`/`description` on conflict, stomping admin team-text edits — contradicting its own comment. Also latent in the generator: `sqlTextArray` (`generate-seed-sql.ts:35-39`) escapes `"` but not `'`, so a category like `Men's` would corrupt the generated SQL. **Fix:** add a unique key for hero slides and upsert on it; switch product colours/images to keyed upserts; drop role/description from the 0005 conflict update; escape apostrophes.

**13. Brand pages have no static product fallback.** `/studio` falls back to the compiled-in `PRODUCTS` when Supabase is empty/unreachable (`app/studio/page.tsx:8-9`), but `app/brands/[slug]/page.tsx:23-26` uses only `getAllProducts()` — during an outage (or a deploy before migrations) every brand page shows "Products Coming Soon" while the brand header renders fine. The fix isn't trivial to bolt on: the seed `Product` type has no `brandSlugs`; the name→slug mapping already exists in the generator (`generate-seed-sql.ts:164-169`) and should be extracted into the lib.

**14. The static fallback resurrects intentionally deleted content.** Fallbacks fire on "empty result," which is indistinguishable from "admin removed it": soft-deleting a seeded brand removes it from `/brands` but its detail URL keeps rendering from the seed (`app/brands/[slug]/page.tsx:28-39`); deactivating all hero slides, all brands, or all four team members resurrects the static versions. Launch-day content removals won't fully stick. **Fix:** for the brand detail page, query without the `is_active` filter and `notFound()` on existing-but-inactive; in general, fall back only on *error*, not on empty.

**15. Quote and contact submissions can hang forever on "Submitting…".** `app/my-quote/page.tsx:56-102` and `components/contact-section.tsx:29-59` `await` the server action with no try/catch — a transport-level failure (network drop, body-size rejection, deploy rollover) rejects the promise, the button stays disabled with no error shown. Same pattern on every admin Save/Upload button (`image-row.tsx:69-78` etc.), where a >10 MB file is rejected by Next *before* the friendly size message can run, so the documented "File is too large" error is unreachable for exactly its intended case. **Fix:** `try/catch/finally` around all action calls; client-side `file.size` pre-check against the 10 MB cap.

**16. The quote endpoint is the site's only server-persisted data and its easiest spam target.** `app/actions/quotes.ts:6-15` has no `.trim()` and no `.max()` on any field (whitespace-only passes `min(1)`; `message` is unbounded), and the global `serverActions.bodySizeLimit: '10mb'` — raised for the image uploader — applies to this action too, so one anonymous request can insert multi-megabyte rows. No rate limit, honeypot, or captcha. (RLS itself is correctly insert-only and status-pinned.) **Fix:** trim + length caps in the Zod schema; add a honeypot field or rate limit.

## Medium — half-built features and launch hygiene

**17. SEO is essentially absent.** Only the root layout exports metadata — every public page shares one title/description; `/brands/[slug]` has no `generateMetadata` (a Patagonia page is titled "PromoShop Inc | Promotional Products"). No `metadataBase`, no OpenGraph/Twitter cards, no canonical URLs, no sitemap anywhere (and no `Sitemap:` line in robots.txt). The homepage has **no h1** (the hero statement is `<p>` tags, `app/page.tsx:63-74`), and `layout.tsx:42` ships `generator: 'v0.app'` on every page. **Fix:** `metadataBase` + title template in the layout, per-page metadata, `app/sitemap.ts`, an h1 on the home hero, drop the generator tag. **CLIENT** must confirm the canonical production domain (hardening doc says promoshopinc.com vs promoshopstudio.com is still unconfirmed).

**18. The footer newsletter form is fake.** `components/footer.tsx:32-37` shows "Thanks for subscribing!" and discards the email — no action, no table. Real users will believe they subscribed. **Fix:** persist it (the quotes action is a template) or remove the form.

**19. Placeholder/dead UI shipped on every page.** Four social icons `href="#"` (`footer.tsx:25-30`) — **CLIENT** must supply real URLs or they should be removed; "Forgot password?" → `#` and a "Remember me" checkbox wired to nothing (`sign-in/page.tsx:118-122`); the header "Wishlist" heart links to the quote cart — there is no wishlist (`header.tsx:83-86`); sign-up promises wishlist, order history, and personalized recommendations, none of which exist (`sign-up/page.tsx:87-92`); footer "Collections" shows six links that all land on the same unfiltered `/studio` (no query-param filter support; "Tech"/"Eco-Aware" don't even match real category names) (`footer.tsx:134-144`). **Fix:** trim claims to what's real, remove or wire each control, support `/studio?category=` if keeping the collection links.

**20. There is no sign-out anywhere.** `AuthProvider` defines `signOut` but no component calls it; once "signed in," the only auth affordance (Login/Register) disappears and nothing replaces it. **Fix:** account indicator + sign-out in the header.

**21. `pnpm lint` is completely broken.** `package.json` declares `"lint": "eslint ."`, but `eslint` is not a dependency and no eslint config exists in the repo — a fresh clone fails with command-not-found; here it errored on missing flat config. The documented QA gate does nothing (and would have caught e.g. the unused `BrandLogo` import in `app/brands/page.tsx:5`). **Fix:** add `eslint` + `eslint-config-next` and a flat `eslint.config.mjs`.

**22. The only go-live runbook is for a stack that doesn't exist, and the docs contradict the code.** `docs/runbooks/production-go-live.md` walks through an **Azure** launch (App Service, Bicep, GitHub Actions OIDC, MSAL auth, `/api/health`) — none of which exist; the real stack is Vercel + Supabase, so there is currently no valid launch checklist with sign-off criteria that can pass. README claims "no admin UI" while documenting `/admin-dashboard`, says there are three migrations (there are five), and its Vercel env instructions omit `SUPABASE_SERVICE_ROLE_KEY`. `docs/hardening-status.md` lists two items as open that are actually fixed (open-redirect, cart guard) — stale docs mean launch sign-off chases ghosts. **Fix:** archive the Azure runbook, write a short Vercel+Supabase one (env vars, migrations 0001→0005 in order, key rotation, domain), correct the README.

**23. Verify the storage migration against the real Supabase project before launch.** `0002_images_and_products.sql:171-176` creates a policy on `storage.objects`; on Supabase projects created after the 2025 storage ownership lockdown, SQL-Editor DDL there fails with "must be owner," aborting the whole migration including bucket creation. If it fails, create the bucket/policy via the Dashboard and strip those lines (the bucket is public-read anyway). **CLIENT/operator:** confirm during setup.

**24. Quote cart: no dedupe, no cross-tab sync, no header badge.** `addItem` always appends (`lib/quote-context.tsx:138-144`), so adding the same SKU/colour twice produces duplicate quantity-1 lines in the submitted quote; there's no `storage` event listener (AuthProvider has one), so two tabs silently overwrite each other's carts; the header "Get a Quote" CTA shows no item count. **Fix:** merge on `(sku, colour, size)`, add a storage listener, render a count badge.

**25. The CAN/USA locale toggle doesn't do what the code intends.** The `t()` spelling function and its 14-entry dictionaries (`lib/locale-context.tsx:44`, `lib/cms/locale.ts`) have **zero call sites** — Canadian spellings are hardcoded into copy USA users see ("Select your colours", "Catalogue"). The toggle only swaps contact info. **Fix:** wire the visible strings through `t()` or delete the dead dictionary so behavior matches code.

**26. Accessibility gaps that clash with the footer's own ADA statement.** The product modal and lightbox have no `role="dialog"`, `aria-modal`, focus trap, or focus restore — keyboard focus stays on the page behind the overlay (`product-detail-modal.tsx:157`, `product-lightbox.tsx:75`); the hero slideshow auto-advances with hover-only pause (no button/keyboard pause — WCAG 2.2.2); the marquee and shimmer animations ignore `prefers-reduced-motion`; the mobile menu button lacks `aria-expanded`. **Fix:** use the already-shipped Radix dialog (or manual focus management), add a pause button, wrap animations in a reduced-motion media query.

**27. The brand-logo marquee loop math is wrong.** `.animate-scroll` translates `-50%` of the *container* (viewport-width flex block), not the duplicated content strip (`brand-logo-scroll.tsx:70`, `globals.css:81-92`) — the "seamless" loop visibly snaps backward each cycle. **Fix:** add `w-max` to the scrolling flex container, then verify in a browser.

## Low — robustness, dead weight, polish

**28. Theme-override gaps.** Changing `brand.primary` recolors everything except the hardcoded darker hover `#d93e36` on the homepage CTA and header button, and `[#111]` shorthand classes aren't matched by the `brand.dark` override (only `#111111`). Separately, `ThemeVarsProvider` (`components/theme-vars-provider.tsx:13-19`) sets hardcoded default `--brand-*` values as inline styles post-hydration, beating the server-injected admin values — latent (nothing consumes `var(--brand-*)` yet) but it silently breaks the documented opt-in API. **Fix:** add the derived-hover and `#111` selectors to the generator; delete `ThemeVarsProvider`. Defense-in-depth: re-validate hex format inside `themeOverrideCss` before injecting into `<head>`.

**29. No `not-found.tsx`, `error.tsx`, or `loading.tsx` anywhere.** Unknown brand slugs and render errors show Next's unbranded defaults; and since the root layout awaits four Supabase fetches per request with no timeout, a *hung* (not failed) connection stalls every route with no skeleton. **Fix:** branded boundary files; consider a fetch timeout.

**30. Admin robustness batch.** Reads are silently truncated at PostgREST's 1000-row default (`product_images` will hit first); zero-row updates report "Saved" (stale tab edits against deleted rows); `replaceImage` uploads to storage *before* validating `target`, leaving attacker-pathed orphans (`actions.ts:39,73-88`); product name/category/description, brand categories, image alt, slide bg_color, and all sort_orders are create-only (typo fixes require the Table Editor); `createProduct` attaches only one brand though the column is an array; duplicate detection sniffs `error.message.includes("duplicate")` instead of code `23505`; read-max-then-insert `sort_order` races; unbounded unauthenticated storage growth (uploads never pruned, no rate limit). Each is a small fix; none should block launch individually.

**31. Dead weight in the repo.** The entire 58-file `components/ui/` shadcn layer is unreachable (one type-only import), dragging ~30 unused dependencies (all 27 `@radix-ui/*`, `recharts`, `react-hook-form`, `embla-carousel`, `date-fns`, `cmdk`, `vaul`, …) — install/build weight and CVE surface only (`tw-animate-css` IS used; keep it). `styles/globals.css` is an unused v0 duplicate with a different palette; `components/theme-provider.tsx` is imported nowhere; ~1.3 MB of orphaned `public/` images; `tsconfig.tsbuildinfo` (508 KB) is committed and `.gitignore` misses `*.tsbuildinfo` and plain `.env`; package is still named `my-project`; no `packageManager` pin. **Fix:** prune (after deciding whether to adopt `ui/dialog.tsx` for finding 26), `git rm --cached tsconfig.tsbuildinfo`, tighten `.gitignore`.

**32. No security headers.** No `headers()` in next.config, no middleware — for a cookie-less marketing site this is low-impact, but the unauthenticated admin route makes `X-Frame-Options: DENY`/`frame-ancestors 'none'` plus a basic CSP worth ~15 lines.

**33. Minor React anti-patterns.** Parent setState called inside state-updater functions in the lightbox/modal (`product-lightbox.tsx:31-45` — the classic "cannot update while rendering" pattern); `Tile` defined inside `BrandLogoScroll`'s render (remounts ~24 tiles on re-render); the modal's exit-animation classes/300ms timers can never animate because it unmounts instantly.

**34. Small content/code inconsistencies.** `site_images.alt_text` is editable in admin but never rendered publicly; `home.slideshow.1-4` registry keys are never seeded so the dashboard never shows those slots (the in-code comment claiming otherwise is false); dead schema surface (`products.is_featured` never used, `brands.website_url` plumbed but never rendered, 0002's header promises a `brand_assets` table that doesn't exist); admin help text says SVG is accepted but the validator correctly blocks it (text fix); footer email is hardcoded while ContactSection reads the CMS key — admin email edits update one but not the other; `/my-quote`'s submit-enable check omits required `lastName`; each page re-fetches `getSiteContentMap()` the layout already fetched (wrap in `React.cache()`); products of a soft-deleted brand show raw slugs as filter chips; `lib/cms/about.ts` says "Promoshop **Canada Ltd.**" with offices in "Windsor and **Los Angeles**" (site says PromoShop Inc., contacts say Detroit) and ends in a sentence fragment.

## CLIENT-supplied values checklist (you'll handle these manually)

- **Vercel env vars:** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (README's instructions omit the third). Without the first two, the site silently shows static fallbacks and quote submissions fail.
- **Production domain** (promoshopinc.com vs promoshopstudio.com — unconfirmed per hardening doc) — needed for `metadataBase`, sitemap, canonicals.
- **Decision: gate `/admin-dashboard` or consciously accept URL-as-secret** (deferral on record; robots.txt leak in finding 3 should be fixed regardless).
- **Content gaps** (per `docs/hardening-status.md`): 9 of 20 brands lack logos; 29 of 46 `site_images` slots empty; team photos are placeholders.
- **Assets/copy:** four hero slideshow images + About hero; privacy/terms/shipping policy text; real social URLs; Visby CF font license decision; confirm phone numbers (Windsor/Toronto/Detroit) and the "Canada Ltd. / Los Angeles" copy.
- **Ops:** rotate Supabase API keys before go-live (open item); apply migrations 0001→0005 in order and verify the storage policy (finding 23).

## What's genuinely solid

Worth knowing what *doesn't* need attention: the build/type-check is clean; the defensive getter contract is correctly implemented everywhere (a Supabase outage can't 500 the layout); RLS is tight (anon is read-only except the constrained quote insert); the service-role client is properly `server-only` and never reaches the browser bundle; upload validation does real magic-byte sniffing and blocks SVG; the seed SQL is in sync with the seed files; `toSafeRedirect` correctly blocks open redirects; localStorage parsing is corruption-guarded; the theme class-override generator covers every variant/opacity actually used (except the two gaps in finding 28); and Next 16 conventions (awaited params, Suspense around `useSearchParams`) are followed.

**My top-five to action first:** 1 (hero content), 2 (fonts), 3 (robots leak), 4 (broken catalog image URLs), 5 (legal pages) — those are what every visitor or every crawler hits on day one. Findings 6–16 are next, since they break the core quote funnel and the CMS contract the client will rely on.
