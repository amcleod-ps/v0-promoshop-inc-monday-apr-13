## 1. TL;DR

- **"PromoShop" is one promotional-merchandise organization with two public
  faces.** 🟢
  - **PromoShop Inc.** — the U.S. parent/agency brand at **promoshopinc.com**
    (Los Angeles; founded **1998**; minority-owned; agency-style "marketing +
    creative + products"). Tagline: *"Where creativity meets promotion."*
  - **PromoShop Canada Ltd.** — the Canadian arm (Windsor, ON HQ + Toronto),
    whose premium-merch **storefront is promoshopstudio.com** and whose older
    catalog lives at `shopcanada.promoshopinc.com`. Tagline: *"Premium merch for
    premium brands."*
  - They are affiliated, not separate companies: the Canadian catalog is a
    **subdomain of promoshopinc.com**, the U.S. site advertises *"nine offices
    and five satellite sales teams throughout the U.S. and Canada,"* and the
    Canadian "about" page lists a Los Angeles corporate office. 🟢
- **This repository is a rebuild of `promoshopstudio.com`** (the Canadian
  *Studio* storefront), even though it brands itself "PromoShop Inc." The
  clincher: the repo's seeded team roster — **Phil Duym (Owner & President),
  Amy Duquette, Ania Wlodarkiewicz, Alex Cyrenne** — matches the
  promoshopstudio.com team exactly, and the metadata copy, 20-brand roster,
  navigation, and CAN/USA locale toggle all match the live Studio site. 🟢
- **Tech:** Next.js 16 (App Router) + React 19 on Vercel, Supabase
  Postgres + Storage, Tailwind v4. Originally generated by **v0.app**. 🟢
- **Defining architectural idea:** every page renders `force-dynamic` and
  every asset URL is cache-busted with `?v=<updated_at>`, so edits made in
  Supabase (Table Editor *or* the in-app `/admin-dashboard`) appear on the
  next request with no rebuild. 🟢
- **Brand palette (from code):** accent **red `#ef473f`**, near-black
  `#111111`, slate `#373a36`, sky-blue accent `#bde7ff`, plus a fuller
  nature-named `--ps-*` palette in `globals.css`. The exact live-site hex
  values could **not** be extracted (see §7); one automated fetch hallucinated
  a navy code, which was discarded. 🟢 (code) / 🔴 (live hex)

## 4. The codebase — back end

**Stack:** Next.js 16 App Router · React 19 · Supabase (`@supabase/ssr`,
`@supabase/supabase-js`) · Vercel · Tailwind v4. Package is `my-project`
(v0.app default). No test framework; `pnpm build` is the de-facto gate.

### 4.1 The render contract (the core idea) — 🟢

`app/layout.tsx:19` sets `export const dynamic = 'force-dynamic'`, so **every**
route fetches fresh on every request. The layout `Promise.all`-loads four
Supabase-backed maps (images, content, team, theme) and wraps the tree in
context providers. Asset URLs carry `?v=<row.updated_at>`; write actions call
`revalidatePath('/', 'layout')`. Net effect: a Table-Editor or
`/admin-dashboard` edit is live on the next request, busting browser/CDN/
`next/image` caches automatically. **Do not** add static generation or image
optimization to content paths — it breaks this contract.

### 4.2 Two-layer data model — 🟢

- **Static seed = source of truth + fallback.** `lib/seed-data/*.ts`
  (`brands.seed.ts`, `products.seed.ts`) and `lib/cms/team.ts`; re-exported by
  `lib/brands.ts` / `lib/products.ts` for types and as a render fallback when
  Supabase is unreachable. `/my-quote` reads static `PRODUCTS` directly.
- **Supabase = live override.** `lib/supabase/*.ts` serve live data to Server
  Components (home, `/studio`, `/brands/[slug]`).
- **`scripts/generate-seed-sql.ts`** compiles the seed files + team roster into
  `supabase/migrations/0003_seed_data.sql` (idempotent; **never hand-edit it**).
  Rerun after editing seeds.

### 4.3 The "registry → provider → resolver with fallback" CMS pattern — 🟢

Four parallel registries share one shape; learn one and you know all four:

| Concern | Table | Fetch (server) | Provider | Read API |
| --- | --- | --- | --- | --- |
| Images | `site_images` | `getSiteImagesMap` | `SiteImagesProvider` | `<SiteImage>` / `useSiteImageUrl` (client); `resolveSiteImageUrl` (server) |
| Text | `site_content` | `getSiteContentMap` | `SiteContentProvider` | `useSiteText` (client); `resolveSiteText` (server) |
| Team | `team_members` | `getTeamMembers` | `TeamMembersProvider` | `useTeamMembers()` |
| Theme | `site_theme` | `getSiteThemeMap` | injected `<style>` in `<head>` | Tailwind class overrides + `var(--brand-*)` |

Conventions: stable dotted keys (`site.logo`, `brand.<slug>.logo`,
`team.<slug>`); helpers in `lib/image-registry.ts`; **every getter swallows
missing-env/errors** and returns `{}`/`[]`/defaults so a Supabase outage never
500s the layout.

### 4.4 Three Supabase clients — 🟢

| File | Key | Context | Notes |
| --- | --- | --- | --- |
| `lib/supabase/server.ts` | anon | Server Components | cookie-aware (`@supabase/ssr`); default for reads |
| `lib/supabase/client.ts` | anon | browser | minimal use today |
| `lib/supabase/admin.ts` | **service-role** | server-only | `import "server-only"`; bypasses RLS; **only** in `/admin-dashboard` actions |

### 4.5 Database schema & RLS — 🟢

Five hand-applied migrations (`supabase/migrations/`, run in order). A shared
`set_updated_at()` trigger stamps `updated_at` on every UPDATE (this powers
cache-busting).

| Table (migration) | Purpose | Public RLS |
| --- | --- | --- |
| `brands` (0001, +cols 0002) | brand listing/detail; `slug` unique; `categories[]`, `featured` | SELECT where `is_active` |
| `hero_slides` (0001) | homepage slideshow | SELECT where `is_active` |
| `quote_requests` (0001) | inbound quote form | **INSERT only**, where `status='new'` AND `admin_notes IS NULL`; no public read |
| `site_images` (0002) | keyed image registry (`key` PK) | SELECT all |
| `products` (0002) | catalog (`sku` PK; `brand_slugs[]`, GIN index) | SELECT where `is_active` |
| `product_colours` (0002) | colourways (`UNIQUE(product_sku,name)`) | SELECT all |
| `product_images` (0002) | gallery (FK→product/colour, cascade) | SELECT all |
| `site_content` (0004) | editable text (`key` PK) | SELECT all |
| `team_members` (0005) | roster (`slug` PK) | SELECT where `is_active` |
| `site_theme` (0005) | editable palette (`key` PK) | SELECT all |
| Storage `site-images` (0002) | public bucket for all uploads | public SELECT; writes via service-role |

Public has **no write** anywhere except inserting a new quote request. All
admin writes go through the service-role key.

### 4.6 Server actions — 🟢

- `app/actions/quotes.ts` — `submitQuoteRequest()`: Zod-validated insert into
  `quote_requests` (anon client; RLS permits `status='new'`).
- `app/admin-dashboard/actions.ts` — `replaceImage`, `removeImage`,
  `updateSiteContent`, `updateHeroSlideText`, `updateBrandText`. 10 MB image
  cap; uploads to `site-images` with a timestamped path; each calls
  `bumpCaches()`.
- `app/admin-dashboard/create-actions.ts` — create/soft-delete for brands,
  products, colours, product images, hero slides, site images, team members,
  plus `updateThemeColor` (hex-validated). Soft delete = `is_active=false`.
- **Brand-logo dual-write gotcha:** a logo is read from both `brands.logo_url`
  (canonical) and `site_images['brand.<slug>.logo']` (legacy override, checked
  first). `replaceImage`/`removeImage` write **both**; touch one, touch the
  other or the override shadows the column.

### 4.7 `/admin-dashboard` — the live CMS — 🟢

Despite README phrasing ("no admin UI"), `app/admin-dashboard/` **is** a working
CMS with five tabs (Images, Text, Products, Team, Theme), built largely with
inline styles. It has an opt-in Basic-auth gate via `ADMIN_DASHBOARD_PASSWORD`;
when that env var is unset, the URL is the secret. `robots: { index:false,
follow:false }`. Files: `page.tsx` (fetches 9 tables),
`dashboard-list.tsx`, `image-row.tsx`, `text-row.tsx`, `products-tab.tsx`,
`team-tab.tsx`, `theme-tab.tsx`, `create-forms.tsx`. Requires
`SUPABASE_SERVICE_ROLE_KEY`; renders a config-error screen if it's missing.

### 5.5 Interaction & layout patterns — 🟢

Responsive grids (products 2→3→4 cols; forms 1→2 cols), `useIsMobile()` at a
768px breakpoint, Embla carousels (hero + lightbox), Radix dialogs with focus
trap/scroll-lock, sonner toasts, sticky studio filter sidebar, real-time
`useMemo` filtering. Quote flow persists to `localStorage` via
`lib/quote-context.tsx`.

### 5.6 Accessibility — 🟢

Explicit `alt` text throughout (sourced from registry rows), `aria-label`/
`aria-pressed` on the locale toggle and carousel controls, `sr-only` labels,
`aria-hidden` on decorative accents, label/`htmlFor` form associations, and a
footer **WCAG 2.1 A/AA** commitment notice. Radix primitives supply dialog
semantics and keyboard handling.

### 5.7 Client-only state (not security) — 🟢

Customer "auth" (`lib/auth/AuthProvider.tsx`, `/sign-in`, `/sign-up`) is a
browser-local quote profile, not access control. The quote cart
(`lib/quote-context.tsx`) and locale (`lib/locale-context.tsx`, CAN/USA
spelling via `t(key)`) are also **localStorage-only** UI personalization. The
only server-persisted data is quote submissions.

## 9. Why this is a committed markdown file

The request offered "a markdown file **or** a GitHub issue." A committed
markdown file in `docs/` is the more optimal home for *durable context*:

- **Versioned & co-located** with the code it describes; it moves with branches
  and shows up in diffs/PR review when it drifts.
- **Greppable and AI-readable** — future Claude Code / contributor sessions find
  it the same way they find `CLAUDE.md`, which an issue can't offer.
- **Matches existing convention** (`docs/CLIENT-SETUP-AUDIT.md`,
  `docs/cms-answer.md`, `docs/asset-map.md`).

A GitHub issue is better for *discussion/visibility* but isn't version-tracked,
drifts from code, and is awkward to consult while coding. **Recommendation:**
keep this file as the source of truth; optionally open a short tracking issue
that links here if the team wants a discussion thread. (Say the word and I'll
open that issue.)
